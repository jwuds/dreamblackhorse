# SEO Audit (Dream Black Horse)

You audit this specific site and return concrete, file-level fixes. This is a prerendered SPA: static routes + every horse + every blog post are snapshotted to HTML at build time by `vite.config.js`, then served from the committed `dist/` on Vercel.

## Step 0 — Load context
Read `.claude/context/seo-guidelines.md` and `.claude/context/internal-links-map.md` so recommendations match house rules.

## Step 1 — Pick scope
- **Whole site** (default), or a **single route** (e.g. `/blog/{slug}`, `/horses`), or a **single concern** (canonicals, sitemap, schema, titles).

## Step 2 — Run the checks

### A. Crawlability / prerender coverage (most important here)
- Confirm the page exists as real HTML: check `dist/{route}/index.html` (or `dist/index.html` for `/`).
- `<div id="root">` should NOT be empty — it must contain rendered content. Grep the route's HTML for real words (brand, horse names, headings).
- Are all intended routes prerendered? Compare `tools/getPrerenderRoutes.mjs` output (static + Hostinger products + Supabase blog slugs) against `dist/`.
- New content not showing? → it needs `npm run deploy` (rebuild). Flag this first; it's the #1 cause here.

### B. Titles & meta
- Unique `<title>` per route, 50–60 chars (check prerendered HTML, not just components).
- Meta description present, 140–160 chars, unique.
- Flag routes that fall back to the generic `index.html` title (pages without `SEOHead`/Helmet — e.g. some legal pages).

### C. Canonicals (regression-watch)
- Exactly **one** `<link rel="canonical">` per page, pointing to its own URL.
- This is enforced by `vite.config.js` `postProcess`. If you see duplicates or homepage canonicals on subpages, that's a regression — check that postProcess still runs and that nothing hardcodes a canonical in `index.html` or components.

### D. Structured data
- LocalBusiness JSON-LD present (in `index.html` + `src/App.jsx`).
- Blog posts: BlogPosting + FAQ schema? Product pages: Product schema? If missing, hand off to `schema-markup`.
- Validate mentally against Google's required properties; recommend testing in the Rich Results Test.

### E. Sitemap & robots
- `public/sitemap.xml` (auto-generated by build) includes every public route and matches `dist/`. Confirm count parity.
- `robots.txt` allows `/`, disallows `/admin/`, references the sitemap.

### F. Internal linking
- Each blog post links to `/horses` or a `/product/:id` + 1–2 related posts (per internal-links-map).
- No internal links to admin/auth routes. No broken `/product/{id}` (inventory turns over).

### G. Content / on-page
- One H1 per page; logical heading order.
- Image `alt` text present and descriptive.
- Slugs clean (flag the known leading-hyphen slug `-friesian-horse-for-beginners`).

### H. Performance signals (note, don't over-index)
- Main JS chunk is large (~547 kB) — a known advisory, not blocking. Mention code-splitting as a future optimization only if asked.

## Step 3 — Report
Return a prioritized table:

| Priority | Issue | Where (file/route) | Fix |
|---|---|---|---|
| P0 | ... | ... | ... |

- **P0** = blocks indexing/ranking (empty HTML, duplicate/wrong canonical, missing from sitemap, noindex by mistake).
- **P1** = meaningful (missing/duplicate titles or descriptions, missing schema, thin content).
- **P2** = polish (alt text, slug cleanup, internal-link gaps).

Offer to implement the P0/P1 fixes. For anything requiring a rebuild, remind the user to run `npm run deploy`.
